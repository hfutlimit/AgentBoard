## 21:06 手动排查并修复 API 容器崩溃（根因已定位、已修复）

### 根因（精确）
1. **镜像陈旧 + 迁移链断裂**：`agentboard-api` 镜像（19h 前构建）内 `/app/migrations` 仅含 3 个迁移文件（Branch A: `13053c3b1abd→05e694deb9d6→b39f1d2c7a10`）。而数据库已被宿主机全量迁移升级到 `alembic_version=c4e8a1b2d3f4`（该 revision 在镜像内不存在）。
2. `init_db()` 执行 `alembic upgrade head` 时，`ScriptDirectory` 找不到 `c4e8a1b2d3f4` → 抛 `Can't locate revision identified by 'c4e8a1b2d3f4'`（属 `RevisionError`）。uvicorn lifespan 捕获后 **exit code 3** → `restart: unless-stopped` 无限重启。
3. **数据库 schema 漂移（关键隐患）**：DB 里 `audit_logs/task_dependencies/webhook_configs` 三表已存在（来自 `9f8c2e7d1a4c`，曾被手动/部分执行但版本号未推进），但 `tasks.estimate`、`users.display_name/email/avatar_url` 缺失 → 若直接 `upgrade head` 会试图 CREATE 已存在的表而报 "already exists"。

### 修复步骤（已验证）
1. **数据库版本同步**（一次性容器挂载宿主机 `./migrations`）：
   - `alembic stamp 9f8c2e7d1a4c`（仅更新版本号，不跑 DDL，因三表已存在）→ 避开对已有表的 CREATE。
   - `alembic upgrade head` → 仅补跑缺失的 `d6a1f4e8c2b7`(users 列) 与 `e1f2a3b4c5d6`(tasks.estimate)。
   - 补建 `9f8c2e7d1a4b` 的 6 个性能索引（`IF NOT EXISTS` 等价，原本被 stamp 跳过）。
   - 验证：`alembic_version=e1f2a3b4c5d6`(头)，四列/索引齐备。
2. **让运行容器识别全量迁移**：受 Docker Desktop 无法联网拉取基础镜像（`python:3.13-slim` metadata 获取超时）限制，无法 `docker compose build` 重建镜像。改用项目既有 dev 模式做法，在 `docker-compose.yml` 的 `api` 服务挂载 `./migrations:/app/migrations:ro`（已加注释：网络恢复后重建镜像并移除该挂载）。
3. `docker compose up -d --force-recreate api` 重建容器。
