## BugFix: MCP 端口 8001→18001（22:13）

### 问题
WorkBuddy MCP 管理面板显示 `agentboard` 失败：
- 错误：`streamableHttp connect failed: fetch failed; SSE error: connect ECONNREFUSED ::1:8001, connect ECONNREFUSED 127.0.0.1:8001`
- mcp.json 写的是 `http://localhost:8001/mcp`，但宿主机 8001 端口无响应

### 排查过程
1. **docker 状态**：容器 `agentboard-mcp-1` 正常运行（27h 前创建），内部 uvicorn 监听 0.0.0.0:8001
2. **端口映射**：容器 HostConfig 配置了 `8001:8001` 映射，但 NetworkSettings.Ports = `[]`（绑定失败）
3. **Windows 端口保留**：`netsh interface ipv4 show excludedportrange protocol=tcp` 显示 `7921-8020` 和 `8021-8120` 被排除
4. **实测**：`Test-NetConnection` / PowerShell `TcpListener` 测试 8000-8082 整段全部 DENIED（被 Windows 系统保留给 Hyper-V 动态端口）
5. **历史**：7/12 08:33 mcp.json 修改为 8001 时端口可用（Hyper-V 端口范围是动态分配的），现在被新分配的 8000 段覆盖

### 修复
- `docker-compose.yml` mcp 服务 ports: `8001:8001` → `18001:8001`（容器内 8001 不变，宿主机改 18001）
- `C:\Users\jason\.workbuddy\mcp.json` agentboard url: `http://localhost:8001/mcp` → `http://localhost:18001/mcp`
- `docker compose up -d --force-recreate --no-deps mcp` 重建容器
- 验证：`curl http://localhost:18001/mcp` 返回 200 + `serverInfo: AgentBoard 3.4.4`

### 端口时间线（AgentBoard）
| 时间 | 事件 | 端口 |
|------|------|------|
| 7/11 09:35 | commit `ac4195dc` 引入 MCP HTTP 模式 | 8001（mcp 容器内） |
| 7/12 08:33 | mcp.json 创建，写 8001 | 8001 → 8001（生效） |
| 7/13 11:02 | agentboard-mcp-1 容器创建 | 8001:8001（实际未绑定） |
| 7/13 15:39 | commit `2587a44` API_URL 默认 8000→58124 | API 端口 58124 |
| 7/14 13:35 | agentboard-api-1 容器重建 | 58125:8000（人工端口） |
| 7/14 22:13 | 本次修复 | mcp 宿主机 8001→18001 |

### 踩坑总结（跨项目可用）
- **Windows Hyper-V 动态端口保留会动态变化**：8000 段某天可用不代表永远可用
- **netsh 显示范围 ≠ 实际范围**：netsh 显示 `7921-8020/8021-8120`，实测整个 8000-8082 段都被拒
- **docker 端口绑定失败的特征**：`ports are not available: listen tcp 0.0.0.0:8001: bind: An attempt was made to access a socket in a way forbidden by its access permissions`
- **MCP 连接失败的根因判断**：
  - 容器内 netstat 显示在跑 ≠ 宿主机可访问（端口映射可能失败）
  - `NetworkSettings.Ports: []` 表示端口映射没生效
  - 需要 force-recreate 才能让新 ports 配置生效（普通 up -d 不重新创建容器）
- **避坑**：高段端口（≥8500）通常不被保留，建议 MCP/测试服务用 18xxx/28xxx 等高段
