# 变更提案：前端持续优化（模仿 Jira，小步迭代）

## 背景
AgentBoard 的 SPA（`agentboard/web/static/`：`index.html` / `app.js` / `style.css`）已实现项目树浏览、增删改、状态流转、spec 编辑等基础能力，但交互与视觉仍偏"最小可用"：树形列表 + 表单形态，缺少 Jira 式的状态色、看板、行内编辑、详情抽屉、快捷键等。

## 目标
建立一个**长期、低风险的持续打磨轨道**：在不做破坏性重构、不改动后端契约/数据模型的前提下，持续向 Jira 的交互密度与视觉语言靠拢。

## 非目标
- 不引入新框架 / 构建链 / 包管理器（保持纯 HTML/CSS/JS + fetch）。
- 不擅自改动 `models.py` / `api.py` 契约。
- 不做一次性大型重构；任何大改动拆回小项。

## 范围
- **纯前端打磨**：状态色、看板只读视图、行内编辑、详情抽屉、搜索、骨架屏、深色模式、响应式、快捷键等（Backlog A）。
- **需后端字段的 Jira 式能力**（标签、负责人、截止日期、拖拽排序、评论）记入「需后端」分组（Backlog B），单独评估，不混入小优化。
- 详细 backlog 与迭代规则见 `docs/tasks.md` Epic 11（权威工作清单）。

## 影响
- 仅 `agentboard/web/static/` 下的 `index.html` / `app.js` / `style.css`。
- 不触及数据模型、API、MCP、迁移。

## 退出标准（轨道层面）
- 形成可长期运转的"取一项 → 实现 → 验证 → 勾选 → commit"闭环；auto-task 可无歧义认领下一项。
- Backlog A 大部分落地后，可重新评估是否进入 Backlog B（需后端）阶段。
